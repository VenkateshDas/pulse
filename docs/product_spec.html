Zero subprocessesAll data via kernel APIs only: host_processor_info, host_statistics64, libproc, IOKit, sysctl. No shell invocations in the hot path.
Performance budgetMenu bar + closed window: <1% CPU, <50MB phys_footprint. Measure with ps -o cputime delta over 60s on a quiet system, not %cpu (lifetime-decayed).
No data-driven animationsNever .animation(_,value:) on @Observable state. Each frame re-runs the full NSHostingView layout. Springs only for user-initiated transitions.
Render only when visibleGate all snapshot publishing on visibleViews > 0 AND screen unlocked. Lock screen does NOT flip NSWindow.occlusionState — use com.apple.screenIsLocked DistributedNotification.
Safety Vault firstEvery file deletion, without exception, stages to SafetyVault before removal. Nothing is ever directly unlinked.

Layout-stable numbersAll dynamically updating numbers use String(format:) with fixed width + monospaced font. Text intrinsic size must never change between samples or layout thrashes.

⑦ Honest by design — brand position

3.1 · Always-On Menu Bar

shipped v0.2 replaces iStat Menus

Why

iStat Menus has 505K downloads. The menu bar is where Mac power users live — it's the habit anchor. A persistent, glanceable display turns Pulse from a tool you open once into something running 24/7. That daily presence is the distribution moat: the user sees the Pulse icon every hour, which means every time their Mac slows down, Pulse is already there. Stats has 38.7K GitHub stars doing this for free — we need to be meaningfully better, not just equivalent.

What

  • Configurable items: CPU (E/P-core split on Apple Silicon), memory pressure, disk free, network ↑↓, temperature, battery — each independently toggleable with compact text / graph / icon styles
  • Popover on click: 30s sparklines for all vitals, top 3 processes by CPU and memory, disk free with 7-day delta ("−4.2 GB this week"), quick actions (open Command Center, run Quick Clean, Empty Trash with size preview)
  • Smart alerts — all actionable, all rate-limited, all individually dismissable: disk < threshold, sustained memory pressure, runaway process >90% CPU for 5 min, app preventing sleep, Time Machine overdue, battery health change
  • Performance: <1% CPU, <50MB RSS — published on the website and enforced in CI

Impact

  • Habit loop: user checks menu bar dozens of times/day → Pulse becomes part of their mental model of their Mac
  • Alert-to-revenue funnel: "disk < 10 GB" alert → opens Command Center → cleans → loves the product → writes review
  • Benchmark anchor: 0.47% CPU / ~30MB at idle (measured on M2) gives a concrete marketing claim that Stats can't match on features

When

Persistent from first launch. The menu bar item is the first thing the user sees and the last thing they'd uninstall. Every alert fires in a moment of genuine need, not as noise — rate-limiting and progressive dismissal prevent alert fatigue. The popover is the 5-second "is my Mac okay?" check that replaces Activity Monitor.

3.2 · Dashboard — Vitals + Cause→Fix

shipped v0.2 UNIQUE: cause→fix

Why

Every monitoring app shows numbers. None of them explains what to do about them. The gap is enormous: iStat Menus shows a red CPU graph; the user's only option is to open Activity Monitor, scroll around, guess, and force-quit something randomly. That's 2010 UX. Pulse closes this by linking every alert card directly to its cause and providing a one-click fix. This turns monitoring from a spectator sport into a productive action.

What

  • 4 vital cards: CPU (E/P split, 24H gap-honest history line), Memory (pressure-colored ring, swap usage), Disk (free space, 7-day growth delta), Thermals (SMC CPU/GPU avg temps, system watts via PSTR)
  • Cause→Fix alert cards below vitals: each maps a metric threshold to its top offenders + a button. Examples: CPU hog → process name + Quit; Memory pressure → swap offenders + Quit; Disk critical → jump to Storage; Sleep blocked → app name + Reveal
  • Top-processes panel: 5 processes by CPU, scaleEffect bars (render-only, zero layout thrash)
  • CPU 24H history chart: gap-honest (nil = gap, isolated point = dot, never interpolate); persists to cpu-history.json, survives sleep correctly
  • Greeting with uptime, system load — human-readable context, not raw numbers

Impact

  • Removes the "now what?" moment — the single biggest pain point in monitoring apps
  • Time-to-resolution drops from minutes to seconds for the 80% of common Mac slowdowns
  • Trust builder: user sees Pulse correctly identify the problem → trusts it for cleaning decisions
  • Gap-honest history is a marketing claim: iStat and Stats both interpolate across sleep gaps, showing false trends

When

Primary landing screen when the user opens Command Center. Cause→Fix cards appear only when thresholds are crossed — not as permanent noise. The 24H chart is most valuable the morning after a heavy workday or overnight backup, when the user wants to understand what happened while they slept.

3.3 · Storage Map

shipped v0.5 replaces DaisyDisk UNIQUE: lenses + APFS-correct

Why

DaisyDisk has a 9.99 price point and a beautiful sunburst — but it shows space, it can't explain it, and it offers no safety guidance. Users who click a folder in DaisyDisk still have to manually decide if it's safe to delete. The "lenses" concept — overlaying safety, age, and ownership on the same visual — turns a beautiful display into an actionable one. APFS-correct math solves the "Finder says 50GB free but I can't use it" confusion that generates support tickets for every Mac utility.

What

  • Treemap (default) + sunburst (toggle) of any volume or folder, animated zoom, 60fps
  • Overlay lenses on the same map: Safety lens (green/yellow/red tint by deletability), Age lens (heat by last-accessed), Owner lens (dynamic color by actual category) — switch with a segmented control, no re-scan
  • APFS-correct: Explains the free-space mystery with explicit breakdown: Used vs Purgeable vs Free vs Total. (Note: Purgeable reclamation is handled natively by macOS; zero subprocesses or dummy files used).
  • Scan engine: parallel getattrlistbulk walker, depth-6, ~1M files in <30s, instantaneously sized with size streaming without fastDirectorySize blocking. Incrementally updated via FSEvents.
  • Click any cell → detail panel: size, explicit (non-ghost) item count, owning app, last-modified, safety grade, action buttons

Impact

  • Safety lens is a demo moment — turns the map from art into a decision tool in one click
  • Incremental FSEvents updates mean the map stays live after cleaning, showing reclaimed space instantly — vs DaisyDisk requiring a full re-scan
  • APFS-correct math resolves a category of confusion no competitor addresses; becomes a specific review talking point

When

User navigates to Storage tab, or taps "See what's using space" from a low-disk alert. First scan streams results progressively (no spinner wall). Subsequent opens are instant due to FSEvents cache. The safety lens is most impactful right before a user first clicks "Clean" — it builds the visual foundation for trusting the Smart Clean results.

3.4 · Smart Clean — Evidence-Based

shipped v0.5 UNIQUE: per-item evidence + safety grades

Why

CleanMyMac's top complaint is exactly this: "it deleted something I needed" and "I don't know what it's doing." Their UI says "4.2 GB of System Junk" with a single Clean button. Blind trust is the product. Pulse inverts this. The cleaning UI shows every item in plain English with its consequence — before anything is touched. This isn't just a UX choice; it's a brand position. Evidence-based cleaning is the thesis, and Smart Clean is where that thesis is most visible.

What

  • Categories: app caches (per-app rows), logs, browser caches, old downloads & installers (>30d), Xcode/dev junk (when Dev Mode on), mail attachment cache, iOS backups, language packs, Trash, local Time Machine snapshots, purgeable space
  • Every row: plain-English description, owning app icon, last-used date, size, safety grade (SAFE / CAREFUL / REVIEW), consequence sentence ("Spotify will re-download artwork as you listen"), expandable file list, Finder reveal
  • Default selection = SAFE tier only. CAREFUL requires explicit per-item tick. REVIEW items are never bulk-selectable — Finder-first workflow enforced
  • Post-clean: freed space verified from real volumeAvailableCapacity delta, not estimates
  • Space forecast: "At this week's rate, full in 23 days" with trend line
  • All deletes → Safety Vault. No exceptions.

Impact

  • Zero "I didn't mean to delete that" moments — Vault catches the mistakes; evidence prevents the reckless ones
  • Trust compounds: each successful, verifiable clean makes the next one easier and the review stars higher
  • Conversion driver: the first clean is the aha moment of the trial period — must show real savings quickly
  • Actual freed space (df delta) vs CleanMyMac's estimated scans = a concrete benchmark claim

When

User triggers Smart Clean from the menu bar popover ("Quick Clean"), from a low-disk alert card, or by navigating to the Clean tab. The SAFE tier is pre-selected and visible immediately. The user reviews, optionally expands rows, and clicks Clean. The result screen shows real freed space and a "Vault contains your last clean — undo anytime" message. This is the primary monetized moment of the trial.

3.5 · Safety Vault — The Moat

UNIQUE IN CATEGORY shipped v0.5 core IP

Why

The #1 reason people don't clean their Macs is fear. Not laziness — fear. "What if I delete something important?" CleanMyMac users have entire Reddit threads about files that shouldn't have been deleted. DaisyDisk has no safety model at all. Trash is permanent after Empty. Safety Vault eliminates the fear category entirely: if everything is undoable for 7 days, cleaning becomes as safe as dragging to Trash. This is the single most defensible feature in Pulse — it's not just better UX, it requires genuine APFS knowledge and a well-designed storage model to implement correctly. No competitor has it.

What

  • All deletions moveItem rename to ~/Library/Application Support/Pulse/Vault/<session-uuid>/ — same-volume APFS rename = instant, zero extra space used until purge
  • Per-session JSON manifest: timestamp, item list, sizes, original paths, safety grades. UI presents these items in a flat, modern file-viewer list rather than rigid session groups.
  • 7-day countdown with configurable window (1–30d); purge order: oldest first when disk is tight
  • Restore/Purge: granular item-level selection with multi-select checkboxes. Dynamic header bar morphs based on selected files (count and size). Collision-safe restores. On partial failure, manifest rewrites with stranded items — never silently loses data
  • Vault size always visible in sidebar footer and dedicated Vault tab
  • "Empty Vault" (or bulk purging selected items) is the only irreversible action in the app — gets the one warning modal
  • System Trash integration: shows Trash size/count, Empty Trash action, "files in Trash haven't freed space yet" education

Impact

  • Eliminates the fear barrier — converts hesitant users into cleaners; cleaning rate increases
  • Every Vault restore = a CleanMyMac horror story avoided — ask those users for App Store reviews
  • Word-of-mouth generator: "Pulse accidentally deleted my DerivedData and I just restored it — amazing" is a better tweet than "Pulse is fast"
  • Structural moat: correctly implementing Vault with collision safety, partial-failure recovery, and APFS-correct sizing is non-trivial — took significant effort, hard to copy quickly

When

Every clean operation, automatically — the user doesn't have to think about it. The Vault is explained once in onboarding ("everything you clean is undoable for 7 days") and then works silently. The Vault tab surfaces when the user is curious or when they need to restore something. The countdown timer creates gentle pressure to confirm the clean was good, which is also a re-engagement touchpoint.

UNIQUE IN CATEGORY v1.0 (launch) highest-LTV segment

Why

Every developer's Mac has a silent disk vampire: stale node_modules in abandoned side projects, years of Xcode DerivedData, Docker image layers from experiments, old .venv environments. A fresh audit of a typical developer machine finds 20–80 GB here — but no GUI cleaner touches it. CleanMyMac doesn't know what node_modules is. DaisyDisk shows it but offers no guidance. Developers are exactly the prosumers who pay one-time for tools, share on Hacker News, and become long-term evangelists. Owning this segment with a feature no competitor has is the highest-ROI investment in the product.

What

  • Toggle in Settings: "Developer Mode" — reveals an extra category in Smart Clean
  • Node.js: node_modules/ in projects where package.json last modified >60 days. Shows project name, size, last-modified. Safe to delete (fully regenerable via npm install)
  • Python: .venv/, .virtualenv/, __pycache__/ in stale projects
  • Java/Gradle/Maven: build/, target/, .gradle/ caches
  • Xcode: DerivedData (grouped by project), old device simulators (xcrun simctl list), Xcode caches, old archives in ~/Library/Developer/Xcode/Archives
  • Docker: dangling images, stopped containers' volumes, unused networks (via Docker socket API — no subprocess)
  • Homebrew: formula cache, old versions, orphaned dependencies
  • All grouped by project for context; safety grade CAREFUL (not SAFE) — dev data is precious

Impact

  • 20–80 GB reclaimed on a typical developer machine in first scan — the most dramatic first-run result possible
  • Developer word-of-mouth: HN/Twitter/Reddit posts from developers with screenshots are more valuable than any ad spend
  • Conversion rate driver: a developer who sees "47 GB in stale node_modules across 23 projects" will pay $34 immediately
  • Lock-in via habit: devs run this monthly as part of machine maintenance

When

Surfaced during onboarding ("Are you a developer?") and toggled in Settings. When enabled, the Smart Clean screen gains a Developer section that appears between SAFE and CAREFUL tiers. Scan depth for project detection: home directory depth-6 walk, same engine as SmartScanner — no extra subprocess needed. Most impactful at the first run on a machine that's been active for 6+ months.

3.7 · Storage Timeline

UNIQUE IN CATEGORY v1.0 (launch)

Why

Every storage tool shows the current state. None shows the trend. "My disk went from 80 GB free to 20 GB free in 3 weeks — what happened?" is a real question with no answer in any existing app. DaisyDisk requires re-scanning from scratch every time. Pulse records lightweight daily snapshots (~200KB/day) of the top ~2K directories and can answer this question precisely: "Final Cut render cache grew 48 GB between Monday and Thursday." That moment — clicking a growth spike and seeing exactly what caused it — is the strongest demo screenshot in the product.

What

  • Daily snapshots of top-2K directory sizes stored as compact binary (~200KB/day, ~73MB/year) in ~/Library/Application Support/Pulse/timeline.db (SQLite)
  • Stacked area chart view: per-folder contributions over selectable range (7d / 30d / 90d / 1y)
  • Click any spike → drill down to the exact folder that grew, with a direct link to that folder in the storage map and a Smart Clean action if applicable
  • Space forecast: linear regression on recent growth → "At this rate, full in 23 days" with confidence band
  • Background snapshot via NSBackgroundActivityScheduler — fires while Mac is awake, doesn't block UI
  • Gap-honest: missing days shown as breaks, never interpolated (same principle as CPU history)

Impact

  • Converts a one-off panic clean into a recurring habit — users open Pulse to understand their disk trends, not just to clean
  • Strongest demo moment in the product — one screenshot of a growth spike + cleanup action goes viral
  • Enables informed cleaning: "This 8 GB folder grew this week" is more actionable than "this 8 GB folder exists"
  • Forecast creates urgency without scareware: a factual projection, not a manufactured alert

When

Timeline needs 7 days of data before being useful. During that window, show "Timeline building — check back in 7 days" with a progress indicator. After 7 days, the tab unlocks and becomes the most shared screen in the app. The space forecast is most compelling for laptop users with fixed SSDs approaching capacity — a specific, common user state.

3.8 · Auto Clean Scheduler

shipped v0.6

Why

Most cleaning happens reactively — when the user's disk is critically full. That's the worst time: the user is stressed, they clean hastily, they're more likely to over-delete. Pulse should make maintenance invisible: the SAFE tier runs automatically on a schedule, the user gets a notification summarizing what was freed, and the disk never reaches crisis level. This is the difference between a tool you use in emergencies and one that prevents them.

What

  • Schedule options: daily / weekly / monthly; time preference (night / morning / anytime); auto-clean SAFE tier only without confirmation (configurable)
  • Trigger: NSBackgroundActivityScheduler — survives sleep/wake correctly; fires when Mac is idle
  • Notification on completion: "Auto Clean freed 1.2 GB — 34 items in Safety Vault" with "Review" and "Undo All" actions
  • Clean history log: JSONL append-only at clean_history.jsonl; each entry links to its VaultSession for restore
  • Preview mode: "Next scheduled run will clean: Chrome Cache 450MB, Derived Data 1.2GB" — always inspectable before it runs
  • History tab: date, items, bytes freed, one-click restore of entire session

Impact

  • Turns Pulse from reactive to proactive — the highest-value shift for retention
  • Weekly notification = weekly Pulse touchpoint without requiring the user to open the app
  • Reduces disk-crisis support load: users who auto-clean weekly never hit 5% free

When

Offered during onboarding ("Want Pulse to keep your Mac clean automatically?"). Default: weekly, Sunday 3am, SAFE tier only, with notification. The "Undo All" action in the notification must work even if the user hasn't opened Pulse since the clean — Vault is always available without opening the main window.

3.9 · Process Monitor

shipped v0.7 replaces Activity Monitor

Why

Activity Monitor is a 2003 app that has barely changed. It shows processes but offers no context, no parent-child relationships meaningful to a non-developer, and no direct fixes from the same screen. The combination of process visibility + one-click actions + cause attribution (the runaway Chrome Helper card on the Dashboard) is what turns Pulse's monitor from a data display into a repair tool. Users who open Activity Monitor are already frustrated — Pulse should be the app they open instead, because it also tells them what to do.

What

  • Process list: flat or tree view (parent-child via proc_bsdinfo.pbi_ppid), sortable by CPU / memory / threads / page faults / name / PID, filterable by name
  • Per-process detail card: CPU %, RSS, virtual bytes, thread count, page fault rate, parent name, 60s CPU sparkline history
  • Actions: SIGTERM with confirmation dialog (no force-kill in v1); system processes that fail get feedback
  • Per-interface network rates (en*, awdl excluded): download/upload bytes/sec, overlaid 60s chart
  • Constraint explicitly documented in UI: per-process network breakdown requires private entitlements — interface-level is what's available, stated honestly
  • Lazy pane: MonitorEngine samples only while the Monitor tab is visible — zero cost when hidden
  • proc_name() fallback: returns empty for other-user processes; fall back to proc_pidpath() basename so launchd daemons show real names

Impact

  • Replaces Activity Monitor for everyday use — simpler, faster, more actionable
  • Process tree view explains "why are there 8 Chrome Helper processes" — a question Activity Monitor cannot answer without expert knowledge
  • MonitorEngine at 0.33% CPU per 2s tick (6.7ms wall) — lightweight enough to leave open all day

When

User clicks "What's using my CPU?" from a Dashboard alert card (direct navigation), or opens Monitor tab directly. Process monitor is most useful during a slowdown event — the exact moment Pulse's menu bar alert fires. The tree view is most useful for developers diagnosing their own app's process hierarchy. The network chart is useful during "something is eating my bandwidth" moments.

3.10 · Health Center

shipped v0.8

Why

Battery health and startup item management are two of the most common things Mac users Google. "How do I check my Mac battery health?" and "why does my Mac take so long to start?" drive enormous search traffic. Existing tools either require opening System Settings (buried) or running system_profiler (takes 2s and returns a wall of JSON). Pulse reads IOKit directly in ~2ms and shows 60-day capacity trend, cycle count, and condition in a clean card. Startup item management — especially disabling the drone army of LaunchAgents that accumulates over time — is something no simple tool handles honestly.

What

  • Battery: capacity %, cycle count, condition, charging status, time to full/remaining — all via AppleSmartBattery IOKit registry (~2ms, never system_profiler)
  • 60-day battery capacity trend: BatteryHistoryStore, one reading/day, shows degradation trajectory
  • Daily active hours chart: "how long did I run on battery each day" — powered by pmset log parse on first launch
  • Startup items: user LaunchAgents from ~/Library/LaunchAgents/ — enable/disable via plist rename (.plist ↔ .plist.disabled), takes effect next login. Honest: shows what CAN be managed; global agents are read-only with a lock icon and explanation
  • Performance benchmark: SHA256 5s + 50MB disk write (F_FULLFSYNC) + 256MB memcpy — transparent formula (phase ratios vs M1 baselines × 1000), result ranges: Good >1000, Degraded 500–1000, Throttled <500. Score persisted with delta vs previous run
  • Constraint honesty: third-party Login Items not enumerable (BTM store is private) — stated plainly in the page header

Impact

  • Battery health card answers one of the most-Googled Mac questions in 2ms
  • Startup item management often reclaims 0.5–2s of boot time for users with 15+ LaunchAgents — immediate tangible value
  • Benchmark gives users a number to share and a reason to run Pulse after any hardware concern
  • 60-day capacity trend is a prompt for AppleCare conversation — users thank Pulse for surfacing this before the battery dies

When

Battery health alert fires when capacity drops below 80% (IOKit condition change) or when the user navigates to Health directly. Startup management is most useful right after installing Pulse on a machine that's been used for 2+ years — the LaunchAgent graveyard is always shocking. Benchmark is run on demand or when the user reports "my Mac feels slow" — the score provides a baseline and a conversation starter.

3.12 · Onboarding — Trust First

v1.0 (launch, pre-flight) BLOCKING: TCC stall known issue

Why

The first 5 minutes determine whether a trial converts to a purchase. Two known failure modes: (1) TCC consent — the first storage scan stalls for minutes blocked on Downloads/Desktop/Documents access with no visible dialog in automated screenshot filtering, making it look like a freeze; (2) trust — if the first screen looks like CleanMyMac's scareware ("347 issues found!"), the target user closes the app immediately. Onboarding must solve both: a proper TCC pre-flight that explains what's needed and why, and a first-run experience that demonstrates competence rather than manufacturing urgency.

What

  • Step 1 — Welcome: product tagline, what Pulse can do, what it will never do (no telemetry, no subscription, data stays local). Privacy pledge as a feature, not fine print
  • Step 2 — Full Disk Access: honest ask with direct System Settings deep-link (x-apple.systempreferences:com.apple.preference.security?Privacy_AllFiles). What it enables, what Pulse will never touch (Documents, Photos Library, iCloud Drive, Mail). Degraded mode if declined — clear badges on what's limited
  • Step 3 — Developer? Single toggle. If yes, Developer Junk Mode enabled immediately
  • Step 4 — Auto Clean: offer the weekly schedule with a one-tap accept. Explain Safety Vault inline: "everything you clean is undoable for 7 days"
  • Step 5 — First scan: streams disk map live (progressive, not a spinner wall). Result: "X GB safely reclaimable" with SAFE items pre-selected and evidence visible
  • Target: user completes first clean within 5 minutes with zero fear

Impact

  • TCC pre-flight eliminates the freeze perception — the single biggest technical risk to trial conversion
  • Privacy pledge in step 1 directly addresses the target user's distrust of cleaners — converts skeptics before they can close the app
  • 5-minute-to-first-clean goal is the primary success metric for trial conversion (target: ≥60% of trials)
  • Degraded mode without FDA means Pulse still works for privacy-cautious users — they become monitors-only users, which is still a distribution win

When

First launch only. Must be blocked before any production release — TCC pre-flight is non-negotiable. The stall without FDA can show up in reviews as "app froze during setup" and permanently damage launch-week ratings. Subsequent launches skip onboarding unless FDA was revoked (detect at startup, show a non-modal banner).

3.13 · ⌘K Command Palette

v1.0 (launch) power user layer

Why

The target user — developers, power users — lives in Spotlight and Alfred. A command palette is the expected power layer in any Mac app aimed at this segment. It also surfaces the full breadth of Pulse's features without requiring the user to explore every sidebar section, which increases feature discovery and perceived value of the purchase.

What

  • ⌘K opens a centered search sheet over any screen; Escape closes
  • Commands: navigate to any section, run Quick Clean, open Vault, open specific folder in Storage Map, run Benchmark, toggle Developer Mode, check battery health, show weekly report
  • Fuzzy search across all commands; recent commands pinned at top
  • All navigation commands trigger the same state as clicking the sidebar item directly

Impact

  • Feature discovery: users who never click Health or Dev Mode find them via ⌘K search
  • Speed for power users: ⌘K → "clean" → Enter is faster than any sidebar navigation
  • Makes Pulse feel native and modern — a 2026 expectation for pro Mac apps

When

Available from any screen once the Command Center window is open. Promoted once in a tooltip during first week of use ("Pro tip: ⌘K opens the command palette"). Never shown again.

4 · The Safety System (core IP)

Safety grades are a real, versioned ruleset — not vibes. Every grade is explainable; every rule is logged.

GradeDefinitionExamplesUI Behavior
SAFERegenerated automatically; zero data loss possible; app behavior unchanged after deleteHTTP caches, DerivedData, dangling Docker layers, logs >30dAuto-selected in Quick Clean; bulk ops allowed
CAREFULAlmost certainly unwanted, but user intent possibleOld installers, stale node_modules, app leftovers, downloads >90dListed, never pre-selected; per-item tick required
REVIEWReal user data that happens to be big or oldForgotten videos, old archives, iOS backupsSurfaced for awareness only; no bulk select; Finder-first

Knowledge Base

  • Per-app rules: bundle ID → which dirs are cache vs data, versioned and signed independently of app releases
  • Community-submittable rules with review gate — compounds into durable moat over time
  • Every grade is explainable: hover → "SAFE because: matches com.spotify.client cache rule v12; directory regenerates on launch"
  • KB updates ship via Sparkle feed without requiring a full app update

Hard Floor — Never Touch

  • ~/Documents, ~/Desktop, ~/Photos Library, Mail data, iCloud Drive contents — regardless of rules
  • System-path denylist: anything outside home directory
  • Open-file check before delete: won't clear cache of an app currently writing to it
  • Running-app awareness: won't clean an app that's currently open
  • Vault always: every delete is staged — the hard floor is the second safety net, not the first

12 · Build Log & Decisions

Living section. Last updated: June 13, 2026 — v1.0 launch prep complete. All milestones M0–M8 implemented (notarization pending Apple Developer enrollment).

Key decisions & rationale

DecisionRationale
All builds via make, never raw swift buildCLT 26.5 has broken ManifestAPI (stale *.private.swiftinterface); Makefile applies SWIFTPM_CUSTOM_LIBS_DIR=~/.local/swiftpm-fix + Testing.framework paths
No data-driven animations0.3s spring per 2s sample = ~12% CPU (NSHostingView re-layouts each frame). Springs reserved for user-initiated transitions only. Removing this took idle from 4.7% to 0.47%
Publish-when-visible modelClosed SwiftUI Window keeps NSHostingView alive — observable mutations re-run layout. Gate on visibleViews > 0. Lock screen does NOT flip occlusionState; use com.apple.screenIsLocked DistributedNotification
Memory pressure colors ring, not usage %High usage with normal pressure is healthy macOS behavior. Coloring by usage % would show red on a Mac that's fine. Honesty principle applied to color semantics
SMC via key-space enumeration, not hardcoded listsKey sets differ per chip generation. Enumerate #KEY once at init, classify by prefix (Tp*/Tg*/TB*T/FNum/PSTR), average plausible values (1–125°C window). Verified on M2: CPU ~50°, GPU ~50°, batt matched Stats app, PSTR watts present on fanless Air
Startup toggle = plist rename (.plist ↔ .plist.disabled)launchd only loads *.plist; rename is reversible, sudo-free, takes effect next login. Global agents shown read-only — no sudo theater
Benchmark uses F_FULLFSYNC on disk phaseWithout it, 50MB write "completes" in RAM in nanoseconds — the number is fiction. F_FULLFSYNC forces physical media write, gives a real hardware number
One pmset subprocess allowed on launchTo backfill battery active-hours history via pmset -g log — once, detached, on startup only. Hot path remains zero-subprocess
Ad-hoc signing for dev buildsDeveloper ID + notarization deferred until Apple Developer account enrolled — user action required

Known gotchas — read before building

  • Measure CPU with ps -o cputime delta over 60s on a quiet system. Single ps %cpu is lifetime-decayed. Ambient agent/browser load at 40%+ inflates Pulse readings ~6×
  • proc_name() returns empty for other-user processes (launchd!). Fall back to proc_pidpath() basename. Without this fallback, powerd escapes the sleep-assertion allowlist
  • host_processor_info lists efficiency cores first on Apple Silicon; E-count from hw.perflevel1.logicalcpu
  • Stale dist/Pulse.app shadows fresh builds (macOS resume relaunches it). Always pkill -x Pulse before judging UI changes. Confirm with ps -o comm
  • Release builds inline string literals ≤15 UTF-8 bytes (Swift small-string) — strings | grep can't prove short UI labels are present
  • AppleSmartBattery on Apple Silicon: CurrentCapacity is a percentage (MaxCapacity pinned to 100). Intel: raw mAh — scale by MaxCapacity. AvgTimeToFull/TimeRemaining sentinel = 65535 minutes → map to nil, never render "~1092h to full"
  • ISO8601 JSON date round-trip truncates sub-second precision — compare struct fields, not whole Codable structs with Date, in tests
  • First storage scan stalls minutes blocked on TCC (Downloads/Desktop/Documents) with no visible dialog. Later scans ~2s. Addressed by the onboarding pre-flight (FDA deep-link before the first scan)
  • UNUserNotificationCenter.current() traps from a bare SwiftPM executable (no bundle). Guard every notification call with Bundle.main.bundleIdentifier != nil
  • Dashboard layout is non-scrolling by design — fits min window 1080×720, bottom row stretches. Verify after any card addition
