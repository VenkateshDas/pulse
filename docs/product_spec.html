⚠️ Non-Negotiable Rules (Apply to Every Module)

Zero subprocessesNever spawn a process for data collection. All data must come from kernel APIs (host_processor_info, host_statistics64, libproc, IOKit, sysctl).
Performance budgetMenu bar + dashboard: < 1% CPU, < 50 MB RSS while sampling every 2 s.
No data-driven animationsNever use .animation on @Observable / @State properties. Every animation frame re-runs the entire hosting view's layout. Springs are for user-initiated transitions only.
Render only when visibleNever publish snapshots when all windows are closed or the screen is locked.
Layout-stable textAll dynamically-changing numbers must use monospaced, fixed-width String(format:) so text intrinsic size never changes between samples.
Safety VaultEvery file deletion must be staged in SafetyVault first. Nothing is ever directly removed.
Build via makeAlways use make build, make test, make run. Never call swift build directly.

1 · Positioning & thesis

One sentence: Pulse replaces CleanMyMac + DaisyDisk + iStat Menus + Sensei with a single native app that is safer than CleanMyMac, more visual than DaisyDisk, more actionable than iStat, and cheaper than all three combined — with no subscription.

What incumbents get right

  • CleanMyMac: one-click simplicity, polish
  • DaisyDisk: the sunburst map — joy of seeing space
  • iStat: always-on glanceability, habit loop
  • Sensei: respect for user control

Where they all fail

  • CleanMyMac: subscription fatigue, over-aggressive deletes, scareware-style upsells, black-box “junk” logic
  • DaisyDisk: shows space, can’t explain it, no safety guidance, no monitoring
  • iStat: shows numbers, offers zero actions — “your CPU is hot, good luck”
  • All: four separate $10–120 apps for one problem; none explain why

Pulse’s wedge

  • Explain, then act: every metric links to its cause and a one-click fix
  • Trust as the product: reversible by default, evidence shown before every delete
  • One app, one price, no subscription
  • Developer-aware: the only cleaner that understands node_modules, Xcode, Docker
Target user (v1): “Prosumer Mac owner” — developers, designers, video/photo people, power users with 256–512GB SSDs perpetually full, who currently own 2+ of the incumbent apps or pirate CleanMyMac. They distrust “cleaners,” love evidence, hate subscriptions. Est. 3–5M reachable users. Mass-market consumers are v2 audience via App Store edition.

2 · The 7 differentiators (why people pay)

Features no single competitor has — each one is a review headline.

① Storage Timeline — “what ate my disk this week?”

Every competitor shows a snapshot. Pulse records lightweight daily snapshots of per-folder sizes (top ~2K directories, ~200KB/day) and renders a stacked area chart over time. Click any growth spike → see exactly which folder grew, which app wrote it, and the matching cleanup action. FSEvents-driven, near-zero cost. Nobody has this. It converts the one-off “panic clean” into a recurring habit and the single strongest demo moment.

② Evidence-based cleaning — the anti-CleanMyMac

CleanMyMac says “4.2 GB of System Junk” and asks for blind trust — its top complaint is aggressive deletion. Pulse shows per item: what it is in plain English, which app made it, when it was last touched, what happens if deleted (“Spotify will re-download artwork as you listen”), and a safety grade. Every claim is inspectable — click any row → Finder reveal + file list. Trust is the moat; this is how it’s built.

③ Undo for everything — the Safety Vault

All deletions stage into a Vault (APFS clones where possible — instant, zero extra space on same volume) with a 7-day countdown before permanent removal. One-click restore of any cleanup, even after closing the app. Headline: “The only Mac cleaner with Undo.” Kills the #1 fear that stops people from cleaning at all.

④ Cause → Fix linking (monitoring that acts)

iStat shows a red CPU graph; Pulse shows the graph plus “WindowServer + Chrome Helper (GPU) are 78% of load — 3 tabs are runaway” with kill/restart buttons. Memory pressure → top swap offenders → quit. Disk almost full → jump to top reclaim. Battery drain abnormal → which app prevented sleep last night. Every alert ends in a button, not a shrug.

⑤ Developer Mode — an unserved, high-LTV segment

One toggle reveals a category no GUI cleaner touches: stale node_modules / .venv / target / .gradle in projects untouched 60+ days, Xcode DerivedData & old simulators (xcrun simctl delete unavailable), Docker dangling images/volumes, Homebrew cache, old toolchains. Routinely 20–80 GB on dev machines — proven by mac-monitor’s own scans. Developers are exactly the prosumers who pay one-time for tools.

⑥ Weekly Pulse Report — retention engine

Saturday notification: “This week: disk grew 12 GB (8 GB = Final Cut render cache), worst battery day Tuesday (Teams), 3.1 GB safely reclaimable.” One screen, three actions. This is what makes a one-time purchase feel alive without nagging — and the screenshot users share.

⑦ Honest by design — the brand position

No scareware (“347 issues found!!”), no red badges to upsell, no telemetry without opt-in, no fake scanning animations, never auto-selects “review” items. Local-only data. The category has a trust deficit; Pulse markets against the category’s dark patterns. This is positioning, but it’s also a feature spec: every screen is reviewed against a published “no dark patterns” checklist.

3 · Feature specification by module

🧭 Menu Bar

shipped v0.2replaces iStat Menus
  • Configurable items: CPU (per-cluster E/P), memory pressure, disk free, network ↑↓, temperature, battery — each toggleable, compact text/graph/icon styles.
  • Popover dashboard: click → 30-second sparkline history of all vitals, top 3 processes by CPU and memory, disk free with 7-day delta (“−4.2 GB this week”), quick actions (open Command Center, run Quick Clean, Empty Trash with size).
  • Smart alerts (all actionable, all rate-limited, all off-switchable): disk < threshold, memory pressure sustained red, runaway process > 90% CPU for 5 min, app preventing sleep, Time Machine not backed up in N days, charging-limit/battery-health change.
  • Performance budget: < 1% CPU average, < 50 MB RSS — published in marketing and enforced in CI (the mac-monitor zero-subprocess discipline carries over: one IOKit/SMC sample loop, no shelling out per tick).

💾 Storage — Map, Timeline & Smart Clean

shipped v0.5replaces DaisyDisk + CleanMyMac core

Disk Map

  • Treemap (default) + sunburst (toggle) of any volume/folder; GPU-rendered (Metal/Canvas), animated zoom, 60fps on 1M+ files. Scan engine: parallel fts/getattrlistbulk walker, ~1M files in < 30s, results cached and incrementally updated via FSEvents — rescans are instant, unlike DaisyDisk.
  • Overlay lenses on the same map: Safety lens (green/yellow/red tint by deletability), Age lens (heat by last-accessed), Owner lens (color by owning app). Lenses are the “more visual than DaisyDisk” claim.
  • APFS-correct math: purgeable space, snapshot sizes (tmutil listlocalsnapshots + diskutil apfs), clone-aware sizing — explains the “Finder says 50 GB free but I can’t use it” mystery, with a “reclaim purgeable” action.

Smart Clean (the mac-monitor scan engine, matured)

  • Categories: app caches (per-app rows, not one blob), logs, browser caches, Xcode/dev junk (Developer Mode), old downloads & installers (.dmg/.pkg > 30d), mail attachments cache, large+old files (review tier), iOS backups, language packs, Trash itself, local TM snapshots, purgeable space.
  • Every row: plain-English description, owning app icon, last-used date, size, safety grade, consequence sentence. Expandable to file list. Finder reveal everywhere.
  • Default selection = safe tier only. “Careful” requires explicit tick; “review” items can never be bulk-selected. (Direct answer to CleanMyMac’s over-deletion reputation.)
  • All deletes → Safety Vault (module below). Post-clean report: actual freed space verified against df, not estimated.
  • Storage Timeline (differentiator ①): stacked area chart, per-folder growth attribution, click-spike-to-clean.
  • Space forecast: “At this week’s rate, full in 23 days” with trend line.
  • Duplicates & Similar (v1.1): Hash-based exact dupes (size prefilter → BLAKE3), near-duplicate images via perceptual hash, “keep newest/largest/in-Photos” rules, always Vault-staged.

🛡 Safety Vault

shipped v0.5unique
  • Staging area for every deletion: same-volume moves are APFS renames (instant); shows per-cleanup “sessions” with restore-all or per-file restore.
  • 7-day auto-expiry (configurable 1–30d); expiry only when disk not under pressure is OFF by default — Vault purges oldest first if user runs low.
  • Vault size always visible; “Empty Vault now” with the one irreversibility warning in the app (modeled on mac-monitor’s Empty Trash modal).
  • System Trash integration: Trash size/item count, Empty Trash action, “files you trashed haven’t freed space yet” education — the exact confusion fixed in mac-monitor Round 6, productized.

📊 Monitor (M4)

shipped v0.7replaces process/network panels

Provides per-process CPU, memory, thread counts, and pagefault rates, plus system-wide network interface telemetry.

⚠️ Constraint: Per-process network breakdown is omitted/documented as unavailable. It requires private entitlements (com.apple.private.network.socket-delegate). M4 delivers per-interface network monitoring instead.

❤️ Health (M6)

shipped v0.8battery, startup, benchmark

Provides fast battery health reading via IOKit, launch-agent management without sudo, and on-demand micro-benchmarks. As built (v0.8): login items are intentionally absent — macOS has no public API to enumerate third-party login items (SMAppService manages only the calling app’s own; the BTM store is private), so StartupItem.Kind is userAgent/globalAgent and the UI says so. condition is derived (PermanentFailureStatus ≠ 0 or capacity < 80% → “Service Recommended”) — the registry has no readable BatteryHealth key. Toggle = rename foo.plist ⇄ foo.plist.disabled, applies at next login; global agents are read-only (lock icon, no sudo theater). Added beyond spec: BatteryHistoryStore tracking daily readings of battery consumption (how many hours the battery was used per day) and BenchmarkStore (run history, persisted score with transparent formula: mean of three phase ratios vs fixed M1-class references × 1000). The benchmark card will provide clear baselines (Good/Bad/Worse) to give the user context.

Performance tip: Do not use system_profiler (takes ~2s). Use raw IOKit reads of AppleSmartBattery (takes ~2ms).

🛠 Dev Mode (M7)

shipped v0.9diagnostics & sysctls

Exposes a raw diagnostic console for developers: full SMC sensor reads, a sysctl browser, and per-process file descriptor / thread counting. As built (v0.9): Diagnostic console uses a 3-panel layout refreshing every 5 seconds while visible. SMC key dump iterates the SMC keyspace using dumpAll(), parsing keys with known prefixes. The sysctl browser allows querying system config properties directly via sysctlbyname. Process FDs list enumerates active descriptors using proc_pidinfo(..., PROC_PIDLISTFDS, ...) for own-user processes, ensuring zero subprocesses are spawned.

Features

  • SMC Sensor Dump: Enumerate all readable SMC keys and their current raw values (averaging keys of CPU/GPU/battery temps, fan status, and system wattage).
  • sysctl Browser: Read system configuration properties directly from sysctl() without subprocesses (e.g. hw.logicalcpu, hw.memsize, kern.version).

4 · The Safety System (core IP)

The safety grade is the product’s brain. It must be a real, versioned ruleset — not vibes.

GradeDefinitionExamplesUI behavior
SAFERegenerated automatically; zero data loss possible; app behavior unchanged beyond first-run rebuildHTTP caches, derived data, dangling Docker layers, logs > 30dAuto-selected in Quick Clean; bulk ops allowed
CAREFULAlmost certainly unwanted, but user intent possibleOld installers, stale node_modules, abandoned app leftovers, downloads > 90dListed, never pre-selected; per-item tick
REVIEWReal user data that happens to be big/oldForgotten videos, old archives, iOS backupsSurfaced for awareness; no bulk select; Finder-first workflow
  • Knowledge base: per-app rules (bundle ID → which of its dirs are cache vs data), shipped as signed, versioned data updated independently of app releases. Community-submittable rules with review. This KB compounds into the durable moat.
  • Hard floor: never touch ~/Documents, ~/Desktop, ~/Photos Library, Mail data, iCloud Drive contents regardless of rules; system-path denylist; open-file checks before delete; running-app awareness (won’t clear cache of an app mid-write).
  • Every grade is explainable: hover → “SAFE because: matches com.spotify.client cache rule v12; directory regenerates on launch.”

5 · UX principles & key flows

Principles

  • Glance → Understand → Act → Verify: every screen follows this arc; every action shows its verified result (real df delta, not estimates).
  • One window, sidebar nav (Dashboard · Storage · Clean · Monitor · Health · Vault · Dev), translucent material, native toolbar — must feel like a system app, not a web page.
  • Speed is a feature: launch < 1s, popover < 100ms, map navigation 60fps; first scan streams results progressively (no spinner walls).
  • No dark patterns checklist enforced per screen (no fake urgency, counts, red-badge upsells, pre-selected risk).
  • Keyboard-first power layer (⌘K command palette) — heritage from the TUI, loved by the target user.

Flow: first run (the trust onboarding)

  1. Welcome → ask for Full Disk Access with honest screen: what it enables, what the app will never do, link to privacy page. Works in degraded mode if declined (clear badges on what’s unavailable).
  2. First scan streams the disk map live (delightful, fast).
  3. Result: “14.8 GB safely reclaimable” — safe tier pre-selected only, evidence visible, Vault explained inline (“undoable for 7 days”).
  4. Clean → animated freed-space verification → menu bar item appears → “Pulse now watches so next time takes one click.”

First-run goal: user deletes something within 5 minutes and feels zero fear. That feeling is the product.

13.2 · Decisions taken (and why)

DecisionRationale
SwiftPM only, no XcodeThis machine has Command Line Tools 26.5 only. Full app builds, tests and .app bundling work via make build/test/run/bundle in pulse/. Xcode optional until App Store submission.
All builds go through makeCLT 26.5 install is broken (stale PackageDescription private swiftinterfaces) — Makefile applies SWIFTPM_CUSTOM_LIBS_DIR=~/.local/swiftpm-fix + Testing.framework paths. Raw swift build fails.
HALO motion rule amended: no animations on data-driven valuesEach animation frame re-runs the entire NSHostingView layout. A 0.3s spring per 2s sample = ~12% CPU; removing it = 1.3%. Springs reserved for user-initiated transitions (Details expand, nav).
Publish-when-visible modelSampling always runs (history never gaps); UI snapshots publish only when a view is on screen AND screen unlocked. Closed windows keep live NSHostingViews; lock screen does not change occlusion state (needs com.apple.screenIsLocked distributed notifications).
Memory ring colored by kernel pressure, not usageHigh usage with normal pressure is healthy macOS behavior. Honesty principle applied to color semantics.
Info-severity alerts don’t flip health statusSleep blockers etc. are FYIs; header + sidebar footer only degrade on warning/critical. Avoids alarm fatigue.
SMC read via key-space enumeration, not hardcoded key listsKey sets differ per chip generation. Pulse enumerates #KEY once at init and classifies by prefix (Tp*=CPU, Tg*=GPU, TB*T=battery, FNum/F0Ac=fans, PSTR=system watts), averages plausible values (1–125 °C window). Works across models without per-model tables. ProcessInfo.thermalState remains the fallback when SMC is unavailable.
Quit action = SIGTERM + confirmation dialogPolite quit, no force-kill in v1; system processes fail gracefully with feedback.
Health lists launch agents only, no login itemsNo public API enumerates third-party login items (SMAppService = own app only; BTM store private). Honesty principle: show what can truthfully be read, explain the gap in the page header.
Startup toggle = plist rename (.plist ⇄ .plist.disabled)launchd only loads *.plist; rename is reversible, sudo-free, and takes effect at next login. Global agents shown read-only.
Benchmark score is a transparent formulaMean of three phase ratios against fixed reference speeds (2000 MB/s SHA256, 3000 MB/s disk write, 60 GB/s memcpy ≈ base M1) × 1000. Disk phase uses F_FULLFSYNC — without it a 50 MB write “finishes” in RAM and the number is fiction.
Ad-hoc signing for dev buildsDeveloper ID + notarization deferred until Apple Developer account is set up (user action).
Allowing pmset log subprocess once on launchTo backfill previous days' battery consumption history (awake active hours) on startup, we run pmset -g log in a detached background task once. This does not affect live monitoring performance (the live loop remains zero-subprocess) and populates historical usage immediately.
Dev Mode implemented with zero subprocessesUsed sysctlbyname, proc_pidinfo, and SMC key space enumeration (dumpAll) to fully inspect system state without calling lsof or shell commands, staying well within performance limits.

13.4 · Things to remember (gotchas for future sessions)

  • Measure CPU with ps -o cputime deltas over 60s or top -l 3 — single ps %cpu is lifetime-decayed and startup-polluted.
  • proc_name() returns empty for other users’ processes (root daemons) — fall back to kinfo_proc.kp_proc.p_comm via sysctl. Without this, powerd showed as “pid 570” and escaped the sleep-assertion allowlist.
  • host_processor_info lists efficiency cores first on Apple Silicon; E-count from hw.perflevel1.logicalcpu.
  • Changing text must stay layout-stable: monospaced + fixed-width formats; process bars use scaleEffect (render-only), never frame(width:) per sample.
  • Window frame persists in defaults under com.pulse.app — delete it after changing default window size.
  • Disk growth history persists at ~/Library/Application Support/Pulse/disk-history.json (6h cadence, needs 24h span before showing a rate); CPU minute history at cpu-history.json (written every 5 finalized minutes — quitting sooner loses ≤5 min, accepted).
  • First storage scan can stall minutes blocked on macOS TCC consent (Downloads/Desktop/Documents) before any file I/O returns — subsequent scans take ~2s. Production build needs usage strings + a pre-flight access step in onboarding.
  • Treemap is a ~40-line binary split (size-balanced halves, alternate axis) — good enough at ≤8 cells; revisit squarified + Metal only if cell counts grow.
  • A stale dist/Pulse.app instance can shadow a fresh debug build (macOS relaunches it via resume) — pkill -x Pulse and check ps -o comm shows the binary you think you launched before judging UI changes.
  • The dashboard layout is non-scrolling by design: fits min window 1080×720, bottom row stretches.
  • AppleSmartBattery on Apple Silicon reports CurrentCapacity as a percentage (MaxCapacity pinned to 100); Intel reports raw mAh — scale by MaxCapacity. Capacity health = AppleRawMaxCapacity / DesignCapacity. Gauge “unknown” sentinel is 65535 minutes (AvgTimeToFull/TimeRemaining) — map to nil, never render “~1092h to full”.
  • Battery consumption history persists at ~/Library/Application Support/Pulse/battery-history.json (one entry/day of active hours, 60-day window, backfilled via pmset -g log on startup); benchmark runs at benchmark-history.json (capped 50).

14 · Open Questions & Architectural Decisions

  1. M4: Network per-process breakdown vs interface-only:

    Retrieving per-process network I/O requires the com.apple.private.network.socket-delegate private entitlement or a full Network Extension. Do we pursue this complex avenue or is per-interface telemetry sufficient for our M4 monitoring scope? (Decision: Default to interface-only first, clearly stating the restriction in the UI).

  2. M3: Sandbox model for background scheduling:

    UserNotifications and background activities function most reliably inside a sandbox. Since Pulse is currently unsandboxed (direct distribution, notarized DMG), should we configure a sandbox for M3 or rely on non-sandboxed scheduling triggers? (Decision: Use non-sandboxed helper agents or accept sandbox limits if Setapp requires it later).

  3. Distribution targets (App Store vs direct):

    Confirming if direct distribution (notarized DMG, Homebrew cask) remains primary over MAS. Direct distribution allows unsandboxed execution, which is crucial for full directory scan access, SMC keys, and LaunchAgents control. MAS edition would require sandboxed dual-SKU with limited scope.

  4. Auto-update pipeline:

    Should we integrate Sparkle 2 updater before M3 is shipped to beta testers, or defer it until the health check phase (M6) is completed? (Decision: Defer to keep velocity high, but build ad-hoc package first).

15 · Lessons Learnt & Decisions Taken

  • Sudoless Network Monitoring: Instead of running netstat or similar subprocesses, we implemented a native NetworkSampler using Darwin's getifaddrs API to gather cumulative interface bytes (ifi_ibytes and ifi_obytes). This strictly adheres to the Zero Subprocesses performance constraint.
  • Power Monitoring via SMC: Displaying detailed CPU, GPU, and ANE power splits is not possible without reverse-engineered hardware-specific undocumented SMC keys or spawning powermetrics via sudo. We decided to use the reliable PSTR SMC key which gives the Total System Watts, aligning perfectly with our sudden/heavy metrics philosophy.
  • UI Layout Density (btop-inspired): The Dashboard charts were refactored into a dense 2x2 grid (CPU, Memory, Network, Power) rather than a tall scrolling stack. KPI cards were made compact and uniformly sized. X-axis scale labels (-30m, -15m, LIVE) were moved to sit underneath each individual graph rather than a shared global footer to maximize legibility.
  • UNUserNotificationCenter needs a bundle: Calling UNUserNotificationCenter.current() from a bare SwiftPM executable (no .app bundle) traps at runtime. Every notification call in CleanModel sits behind a Bundle.main.bundleIdentifier != nil guard so make run debug builds degrade gracefully.
  • Stale Pulse.app shadows fresh builds during UI verification: macOS resume relaunched the old dist/Pulse.app next to the new debug binary, showing the pre-M4 sidebar. Always pkill -x Pulse, re-make bundle, and confirm via ps -o comm before judging UI changes.
  • Memory Chart Replacement: We replaced the GPU Temperature chart with a System Memory Utilization chart as GPU telemetry APIs on Apple Silicon are heavily restricted to PrivateFrameworks.
  • proc_name is permission-gated, proc_pidpath is not: proc_name() returns 0 for other users' processes (launchd included), but proc_pidpath() still resolves — MonitorEngine falls back to the executable path's basename so parent names in the detail card stay real instead of "—".
  • One pid-walk per tick, shared across engine methods: MonitorEngine's sample()/tree()/parents()/names() share a collection cached for 500ms, so a UI tick calling all four pays for one proc_listallpids walk, not four. Idle (zero lifetime bytes) en* interfaces are dropped from network telemetry — five dead Thunderbolt rows are noise, not honesty.
  • Generic SMC Dump: Enumerate keys via dumpAll and classify CPU/GPU/battery/wattage prefixes. Keep sysctl and FD lists readable and filterable.
Spec summary: Pulse wins not by having more features than CleanMyMac, but by being the first app in the category where the user never has to trust a black box: every number explained, every delete reversible, every alert actionable, one honest price. The mac-monitor TUI is the validated prototype; PulseKit ports its engine to Swift; launch as “the anti-CleanMyMac.”
